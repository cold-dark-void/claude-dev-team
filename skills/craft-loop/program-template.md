# Loop-Program Template

Canonical skeleton for programs generated by /craft-loop (SPEC-020). The craft
dialogue fills every `<...>` token; the SKILL.md quality checklist must pass
before the file is written to `.claude/loops/<name>.md`. The fenced body below
is what gets copied — never copy this header paragraph into a program.

````markdown
---
name: <kebab-case-name>
target: <loop or goal>
cadence: <dynamic, or a fixed-interval suggestion such as 20m>
status: ready
created: <YYYY-MM-DD>
---

# Objective

<One paragraph: what done looks like, stated in objectively checkable terms.>

# Every iteration

1. Read `.claude/loops/<name>.journal.md`. Recover where the previous firing
   stopped (last entry's `Next` field) and treat any decision card that now has
   an indented `Answer:` line as resolved input. If the journal does not exist,
   create it with a `# Journal — <name>` heading; this is iteration 1.
2. <step>
3. <step>
4. Append a journal entry using the schema below. This is always the last step.

# Stop when

<Objectively checkable condition — a fact of the repo or files, not a judgment
call.> When true: announce "loop complete: <name>", append a final journal
entry, and end the loop — do not continue iterating.

# Never

- Run `git push` or any command that publishes to a remote
- Delete branches or tags, or rewrite git history
- Delete files outside the scope declared in the Objective
- <program-specific guardrails>

# When blocked

If a choice arises that only the user can make: append a decision card under
`Decisions needed` in this iteration's journal entry, then continue with
unblocked work. If no unblocked work remains, append a final entry and end the
loop announcing "loop BLOCKED: <name> — see journal".

# Journal entry schema

Append to `.claude/loops/<name>.journal.md`:

## Iteration <N> — <YYYY-MM-DD>
- Did: <what this firing accomplished>
- State: <where things stand now>
- Next: <what the next firing should pick up>
- Decisions needed:
  - [DECISION] <the question, with enough context to answer it cold>

A card is open until an **indented** `Answer: <text>` line is added beneath it
(leading whitespace required — a bare `Answer:` at column 0 does not close the
card). The answer is written by the user directly, or by a session relaying the
user's decision; the next firing treats answered cards as resolved input. Omit
the `Decisions needed` list when there are none.

Frontmatter `status` values: `ready` (default) or `retired` (manual/format only;
no special runtime path).
````
