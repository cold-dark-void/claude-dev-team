# MEM-002 Spec: Tiered Memory Distillation with LLM Compression

**Date:** 2026-03-14
**Status:** FINAL — all ACs confirmed, ready for implementation
**Ticket:** MEM-002
**Depends on:** MEM-001 (SQLite memory backend, v0.12.0 schema)

---

## Problem Statement

Agent memory accumulates unbounded raw entries. Session-start context loading becomes
noisy and expensive as the raw count grows. There is no mechanism to surface high-signal
memories (key decisions, architectural truths, hard-won lessons) ahead of routine
observations.

## Solution Summary

A 3-tier distillation system on top of the existing `memories` table. Raw writes are
unchanged. An LLM compression pass synthesizes batches of raw memories into digest
summaries, then promotes high-signal digests to a permanent core tier. Session start
always loads core; digest and raw loaded on demand.

A new `/memory-config` command lets users control distillation behavior without raw SQL.

---

## Schema Changes (migration from v0.12.0 → v0.13.0)

```sql
-- Extend memories table
ALTER TABLE memories ADD COLUMN tier INTEGER NOT NULL DEFAULT 0
  CHECK(tier IN (0, 1, 2));
  -- 0 = raw (Layer 0)
  -- 1 = digest (Layer 1, LLM-compressed)
  -- 2 = core (Layer 2, promoted "greatest hits")

ALTER TABLE memories ADD COLUMN archived BOOLEAN NOT NULL DEFAULT FALSE;
  -- TRUE = source row consumed by distillation; excluded from normal queries

ALTER TABLE memories ADD COLUMN distilled_from TEXT NOT NULL DEFAULT '[]';
  -- JSON array of INTEGER memory IDs this row was synthesized from

-- Extend type CHECK constraint
-- (SQLite does not support ALTER COLUMN CHECK; this is enforced by
--  recreating the table or via a migration trigger — implementation detail)
-- New valid type values: 'cortex', 'memory', 'lessons', 'digest', 'core'

-- Config keys added via INSERT OR IGNORE
-- distill_enabled    TEXT  'false'
-- distill_mode       TEXT  'suggest'   (manual | suggest | auto)
-- distill_threshold  TEXT  '50'
-- distilling_lock    TEXT  ''          (empty = unlocked; agent name = locked)

-- schema_version bumped from '1' to '2'
```

---

## Tier Definitions

| Tier | INTEGER | Label | Description |
|------|---------|-------|-------------|
| 0 | 0 | raw | Every memory-store write. Current behavior. |
| 1 | 1 | digest | LLM-compressed batch summaries from raw. |
| 2 | 2 | core | Promoted high-signal memories: decisions, lessons, architecture. |

`archived = TRUE` marks raw rows consumed by distillation. They stay in the DB but
are excluded from all normal queries (`WHERE archived = FALSE`).

## Acceptance Criteria

---

### AC-D1: Schema Migration

**D1.1** Running `/init-team` (or `/init-team --migrate-only`) on a v0.12.0 DB (`schema_version = '1'`) executes the following additions to the `memories` table:
- `tier INTEGER NOT NULL DEFAULT 0` with CHECK `tier IN (0, 1, 2)`
- `archived BOOLEAN NOT NULL DEFAULT FALSE`
- `distilled_from TEXT NOT NULL DEFAULT '[]'`

Each `ALTER TABLE` is idempotent: if the column already exists, the step is skipped without error.

**D1.2** All pre-existing rows in `memories` are updated to `tier = 0, archived = FALSE, distilled_from = '[]'` during migration. Rows inserted after migration receive these defaults automatically.

**D1.3** The `type` column CHECK constraint is updated to accept `'digest'` and `'core'` in addition to the existing `'cortex'`, `'memory'`, `'lessons'`. After migration, `INSERT INTO memories(type) VALUES ('digest')` succeeds.

**D1.4** The following config keys are inserted via `INSERT OR IGNORE` into the `config` table:

| key | default value |
|-----|---------------|
| `distill_enabled` | `'false'` |
| `distill_mode` | `'suggest'` |
| `distill_threshold` | `'50'` |
| `distilling_lock` | `''` |

Existing rows with these keys are not overwritten on re-run.

**D1.5** `schema_version` in the `config` table is updated to `'2'` after all migration steps complete successfully. A DB already at version 2 skips all migration steps without error.

**D1.6** If any migration step fails (e.g., disk full, locked DB), the migration exits non-zero and prints the failing step. `schema_version` is NOT updated to `'2'`. Partial migrations are safe to retry.

---

### AC-D2: `memory-store` — Post-Write Threshold Check

**D2.1** After a successful `INSERT` into `memories`, `memory-store` reads `distill_mode` from the `config` table.

**D2.2** If `distill_mode = 'manual'`: no threshold check. Return immediately after the INSERT.

**D2.3** If `distill_mode = 'suggest'` or `distill_mode = 'auto'`: count `tier = 0 AND archived = FALSE` rows for the writing agent. If count >= `distill_threshold`, print a notice:
```
[memory] @<agent> has <N> raw memories (threshold: <T>). Run /memory-distill to compress.
```
No distillation is triggered. Return after printing.

**D2.4** If `distill_mode = 'auto'`: the same notice is printed AND distillation is scheduled to run at session end via the wrap-ticket hook (see AC-D6.5). Distillation does NOT block the `memory-store` call.

**D2.5** If `distill_enabled = 'false'`: skip the threshold check entirely regardless of `distill_mode`. Zero additional DB queries are executed in this path.

**D2.6** If the threshold check itself errors (e.g., DB locked), `memory-store` logs a warning to stderr and returns successfully. The original INSERT is not rolled back.

---

### AC-D3: Distillation — Concurrency Lock

**D3.1** Before starting any distillation pass (manual or auto), the system attempts to acquire the distillation lock with an atomic write:
```sql
UPDATE config SET value = '<agent>' WHERE key = 'distilling_lock' AND value = '';
```
If `changes()` returns 0 (lock held by another agent), distillation is skipped and prints:
```
[distill] Skipped: distillation already in progress (locked by @<holder>).
```

**D3.2** On successful lock acquisition, `distilling_lock` is set to the agent name. After distillation completes (success or failure), the lock is cleared:
```sql
UPDATE config SET value = '' WHERE key = 'distilling_lock';
```

**D3.3** `/memory-distill --force` clears the lock unconditionally before attempting distillation:
```sql
UPDATE config SET value = '' WHERE key = 'distilling_lock';
```
Prints: `[distill] Stale lock cleared. Proceeding.`

**D3.4** The lock is per-DB (shared across all agents). Only one distillation can run at a time against a given `memory.db`.

---

### AC-D4: Layer 0 → Layer 1 Distillation (Batch Compression)

**D4.1** Distillation for a target agent fetches raw memories in batches. Batch size equals `distill_threshold` (default 50). Rows are ordered by `created_at ASC` (oldest first). Only rows with `tier = 0 AND archived = FALSE` are selected.

**D4.2** For a target agent with 200 raw rows (threshold = 50): distillation produces exactly 4 digest rows, one per batch of 50. The batches do not overlap. If the final batch contains fewer than `distill_threshold` rows (e.g., 12 remaining), it is distilled as a partial batch.

**D4.3** For each batch, the system submits a single LLM prompt containing all batch row contents. The prompt instructs the LLM to synthesize the memories into a concise digest summary. The exact prompt template is an implementation detail; it is not specified here.

**D4.4** The LLM response for each batch is inserted as a new row:
```
agent           = <target agent>
type            = 'digest'
tier            = 1
archived        = FALSE
distilled_from  = JSON array of INTEGER IDs of the batch source rows
content         = LLM response text
```

**D4.5** After a batch digest row is inserted successfully, all source rows in that batch are updated: `archived = TRUE`. This update is atomic with the digest insert (within a single transaction). If the transaction fails, neither the digest insert nor the archive update persists.

**D4.6** If the LLM call for a batch fails (timeout, error response), that batch is skipped. Remaining batches continue. Source rows for the failed batch remain `archived = FALSE`. A warning is printed per failed batch:
```
[distill] Batch 2/4 failed: <error>. Source rows not archived.
```

**D4.7** If an agent has zero `tier = 0 AND archived = FALSE` rows, distillation prints:
```
[distill] @<agent>: no raw memories to distill.
```
And returns without error.

**D4.8** Distillation is scoped to one agent per invocation. When distilling all agents (see AC-D7.1), the single-agent logic runs sequentially for each qualifying agent.

---

### AC-D5: Layer 1 → Layer 2 Promotion

**D5.1** After all Layer 0 → Layer 1 batches for an agent complete (even if some batches failed), the system submits a second LLM prompt containing all `tier = 1 AND archived = FALSE` digest rows for that agent. The prompt applies the following promotion criteria:
- Referenced multiple times → promote
- Lesson from a mistake → promote
- Key architectural decision → promote
- Routine codebase mapping → do not promote

**D5.2** For each digest row the LLM marks as promotable, the system executes:
```sql
UPDATE memories SET tier = 2 WHERE id = <digest_id>;
```
The row is promoted in-place. No new row is created. `archived` remains `FALSE`.

**D5.3** Digest rows not marked for promotion remain at `tier = 1`. They are not archived or deleted.

**D5.4** If no digest rows qualify for promotion, the promotion pass completes silently with no writes.

**D5.5** If the promotion LLM call fails, a warning is printed. Already-created digest rows are not rolled back (they remain at `tier = 1`):
```
[distill] Promotion pass failed: <error>. Digest rows retained at tier=1.
```

**D5.6** Layer 2 promotion runs only after a Layer 1 pass for the same agent in the same invocation. It does not run independently in this release.

---

### AC-D6: Session Start Loading

**D6.1** `memory-recall` called with agent filter and no query (session-start load) returns rows matching:
```sql
WHERE agent = '<agent>' AND archived = FALSE AND tier = 2
ORDER BY updated_at DESC
```
This is the default session-start behavior.

**D6.2** An agent with zero `tier = 2` rows falls back to loading all non-archived rows:
```sql
WHERE agent = '<agent>' AND archived = FALSE
ORDER BY tier DESC, updated_at DESC
```
This preserves backward-compatible behavior for agents that have never run distillation.

**D6.3** `memory-recall` with `--include-digest` loads `tier IN (1, 2) AND archived = FALSE`. This is how agents load digest context on demand.

**D6.4** `archived = TRUE` rows are never returned by `memory-recall` under any flag combination in this release.

**D6.5** Semantic search (`memory-recall` with a query string) searches `tier IN (0, 1, 2) AND archived = FALSE`. Archived rows are excluded from all search results.

---

### AC-D7: `/memory-distill` Command

**D7.1** `/memory-distill` with no arguments distills ALL agents whose raw-memory count (`tier = 0 AND archived = FALSE`) meets or exceeds `distill_threshold`. Agents below threshold are skipped silently. Acquires the global lock once before processing all agents.

**D7.2** `/memory-distill --agent <name>` distills a single named agent regardless of threshold. The caller need not be that agent.

**D7.3** On completion, prints a per-agent summary for each agent processed:
```
Distillation complete:
  @pm        raw archived: 52 → 1 digest | promoted to core: 2
  @tech-lead raw archived: 50 → 1 digest | promoted to core: 0
```

**D7.4** `/memory-distill --status` prints tier stats for all agents. No distillation runs:
```
MEMORY TIERS
══════════════════════════════════════════════════
Agent        raw   digest   core   archived
pm             8        2      1         52
tech-lead     12        0      0          0
ic5           45        1      3         23
══════════════════════════════════════════════════
Auto-distillation: disabled
Distill mode:      suggest
Threshold:         50
Lock:              (none)
```

**D7.5** `/memory-distill --force` clears a stale `distilling_lock` before running (AC-D3.3). All other `/memory-distill` behavior is identical.

**D7.6** If `distill_enabled = 'false'`, `/memory-distill` still executes (manual trigger bypasses the enabled flag) and prints:
```
[distill] Note: auto-distillation is disabled. Running manual distillation.
```

**D7.7** If the DB does not exist or sqlite3 is unavailable, `/memory-distill` prints and exits non-zero:
```
[distill] Error: distillation requires the SQLite memory DB. Run /init-team first.
```

**D7.8** `auto` mode distillation hook: when `distill_mode = 'auto'`, the wrap-ticket skill (or equivalent session-end hook) invokes `/memory-distill` for all agents that received a threshold notice during the session. If no agents reached threshold, the hook is a no-op. The hook runs after all other session-end writes.

---

### AC-D8: `/memory-config` Command

**D8.1** `/memory-config list` prints all distillation-related config keys and their current values:
```
MEMORY CONFIG
═══════════════════════════════════
distill_enabled    false
distill_mode       suggest
distill_threshold  50
distilling_lock    (none)
═══════════════════════════════════
```
`distilling_lock` is displayed as `(none)` when value is empty string, otherwise as the agent name currently holding it.

**D8.2** `/memory-config set <key> <value>` updates a single config key. Only the following keys are settable via this command:
- `distill_enabled` — valid values: `true`, `false`
- `distill_mode` — valid values: `manual`, `suggest`, `auto`
- `distill_threshold` — valid values: positive integers (1–9999)

On success, prints:
```
[memory-config] distill_mode = auto
```

**D8.3** Attempting to set an invalid key prints:
```
[memory-config] Error: '<key>' is not a settable config key.
Settable keys: distill_enabled, distill_mode, distill_threshold
```
No DB write occurs.

**D8.4** Attempting to set an invalid value for a valid key prints:
```
[memory-config] Error: invalid value '<value>' for distill_threshold. Expected: integer 1–9999.
```
No DB write occurs.

**D8.5** `/memory-config set distilling_lock` is rejected with:
```
[memory-config] Error: 'distilling_lock' cannot be set manually. Use /memory-distill --force to clear a stale lock.
```

**D8.6** If the DB does not exist, `/memory-config` prints and exits non-zero:
```
[memory-config] Error: memory DB not found. Run /init-team first.
```

**D8.7** `/memory-config` with no arguments prints usage:
```
Usage: /memory-config list
       /memory-config set <key> <value>
```

### AC-D9: Existing Flow Compatibility

**D9.1** `memory-store` called on a v0.13.0 DB with `distill_enabled = 'false'` executes exactly one more SQL statement than v0.12.0: the INSERT itself. The `tier`, `archived`, and `distilled_from` columns are populated by column defaults. No additional queries run.

**D9.2** `memory-recall` with no tier or archived filters returns `archived = FALSE AND tier IN (0, 1, 2)` — all active memories. This is backward-compatible with callers that do not pass tier filters.

**D9.3** `/memory-search` behavior is unchanged. It operates on `archived = FALSE` rows by default. Archived rows are invisible to all existing search flows.

**D9.4** Embedding rows in `embedding_meta` and `vec_memories_*` for archived source memories are retained. They are not queried because the join through `memories` excludes `archived = TRUE` rows.

**D9.5** `memory-store` and `memory-recall` in `.md` fallback mode are completely unmodified. Tiering is a DB-mode-only feature. Calling `/memory-distill` in fallback mode prints the error in AC-D7.7.

---

### AC-D10: Non-Functional Requirements

**D10.1** `memory-store` with `distill_enabled = 'false'` adds no measurable latency vs. v0.12.0. The post-write config read (D2.5 path) executes at most one indexed `SELECT` on `config`.

**D10.2** A full distillation pass (one agent, 200 raw memories, 4 batches) completes within 60 seconds excluding LLM response latency. DB operations only must complete in under 2 seconds total.

**D10.3** The distillation lock mechanism prevents data corruption under concurrent agent writes. Test: two agents each trigger distillation simultaneously; exactly one acquires the lock; the other skips with the message in AC-D3.1.

**D10.4** No new binary dependencies are introduced. Distillation uses the same sqlite3 CLI already required by MEM-001.

**D10.5** The implementation adds no build step. All new functionality is delivered as markdown command/skill files.

---

## Out of Scope (MEM-002)

- `/memory-search --include-archived` (deferred to follow-on)
- Manual Layer 2 promotion without going through Layer 1 first
- Cross-agent combined distillation (single digest spanning multiple agents)
- Distillation for `.md` fallback mode
- Retention policy or auto-purge of archived rows
- Standalone re-embed command for newly created digest/core rows
- Per-agent distillation threshold (threshold is global in this release)
