# MEM-001 Spec: SQLite + Vector Embeddings for Agent Memory

**Date:** 2026-03-14
**Status:** FINAL â€” ACs confirmed, ready for implementation
**Ticket:** MEM-001

## Problem Statement

Agent memory is flat markdown files with no semantic search, no consolidation,
and no cross-agent visibility. Agents cannot find relevant context unless they
know exactly which agent wrote it and where.

## Solution Summary

Replace per-agent `.md` memory files with a single SQLite database per project
(`memory.db`) backed by sqlite-vec for vector search and sqlite-lembed for
air-gapped embedding generation. Add `/memory-search` (semantic) user command
and `memory-store` / `memory-recall` agent skills.

## Platforms Supported

| Platform | Architecture |
|----------|-------------|
| Linux | x86_64, aarch64 |
| macOS | x86_64 (Intel), aarch64 (Apple Silicon) |

Windows not supported. WSL2 = Linux (supported).

## Database Schema

```sql
CREATE TABLE memories (
  id           INTEGER PRIMARY KEY,
  agent        TEXT    NOT NULL,
  type         TEXT    NOT NULL CHECK(type IN ('cortex','memory','lessons')),
  content      TEXT    NOT NULL,
  metadata_json TEXT,
  created_at   TEXT    NOT NULL,  -- ISO-8601 UTC
  updated_at   TEXT    NOT NULL   -- ISO-8601 UTC
);

CREATE TABLE memory_config (
  key   TEXT PRIMARY KEY,
  value TEXT NOT NULL
);

-- sqlite-vec virtual table (created only when extension loaded)
CREATE VIRTUAL TABLE memory_embeddings USING vec0(
  id        INTEGER PRIMARY KEY,
  memory_id INTEGER NOT NULL,
  model     TEXT    NOT NULL,
  dimensions INTEGER NOT NULL,
  embedding FLOAT[384]  -- dimension varies by model; see AC-6
);
```

Note: `context.md` files are excluded from the DB. They remain per-worktree
ephemeral `.md` files as today.

## Embedding Tier Strategy

```
if ollama running (probed once at /init-team) AND has embedding model
  â†’ mode=ollama   (nomic-embed-text, 768-dim)
else if sqlite-lembed + GGUF present
  â†’ mode=lembed   (all-MiniLM-L6-v2, 384-dim)
else
  â†’ mode=fallback (no embeddings, keyword search only)
```

Mode is stored in `memory_config` as `key='embedding_mode'`.

## Acceptance Criteria

### AC-1: Database Initialization

**1.1** Running `/init-team` on a project with no prior memory creates
`.claude/memory/memory.db` at `$MROOT` (resolved via `git rev-parse
--git-common-dir`).

**1.2** The `memories` table exists with columns and constraints exactly as
specified in the schema above.

**1.3** The `memory_config` table exists and accepts arbitrary key-value rows.

**1.4** WAL mode is active: `PRAGMA journal_mode;` returns `wal`.

**1.5** Running `/init-team` a second time does not drop, recreate, or truncate
existing tables or rows (idempotent).

**1.6** If `.claude/memory/` does not exist, `/init-team` creates it before
creating `memory.db`.

**1.7** `memory.db` is added to the project `.gitignore`. If `.gitignore` does
not exist, it is created. If `.gitignore` already exists, the entry is appended
only if not already present.

---

### AC-2: Binary Download

**2.1** `/init-team` detects the host OS and architecture using `uname -s` and
`uname -m` and selects the correct asset names for sqlite-vec, sqlite-lembed,
and the GGUF model.

**2.2** Assets are downloaded to:
- Extensions: `.claude/memory/extensions/`
- Model: `.claude/memory/models/`

**2.3** If an asset already exists at the target path AND its SHA-256 checksum
matches the expected value (sourced from the GitHub release manifest or a
hardcoded table per version), the download is skipped.

**2.4** If a download fails, `/init-team` prints:
```
ERROR: Failed to download <filename> from <url>
Fallback mode (keyword search only) will be used until this is resolved.

To install manually:
  1. Download <filename> from <url>
  2. Place it at <target-path>
  3. Re-run /init-team
```
The error does not abort other setup steps (DB init continues).

**2.5** Unsupported platform (e.g., native Windows) prints a clear error:
```
ERROR: Platform <OS>/<ARCH> is not supported.
Windows users: run Claude Code inside WSL2 (treated as Linux).
```
and exits without attempting any download.

**2.6** The extensions directory and model directory are included in `.gitignore`
(covered by the `memory.db` exclusion or separately â€” both must be excluded).

---

### AC-3: Embedding Mode Detection

**3.1** During `/init-team`, after binary download, the system probes ollama by
sending `GET http://localhost:11434/api/tags` with a 2-second timeout.

**3.2** If the probe succeeds and the response JSON contains at least one model
name that includes the substring `embed`, mode is set to `ollama`. The first
matching model name is stored in `memory_config` as `key='ollama_model'`.

**3.3** If the ollama probe fails (timeout, connection refused, or no embed
model found), and both the sqlite-lembed extension file and the GGUF model file
are present at their expected paths, mode is set to `lembed`.

**3.4** If neither condition is met, mode is set to `fallback`.

**3.5** The resolved mode is stored in `memory_config` as
`key='embedding_mode'`, `value` one of: `ollama`, `lembed`, `fallback`.

**3.6** Running `/init-team --refresh` or `/memory-search --status` re-runs the
mode detection probe and updates the `memory_config` row. It does NOT
re-download binaries unless `--refresh` is combined with a re-download flag
(out of scope for this ticket).

**3.7** `/memory-search --status` (with no query) prints:
```
Memory DB: .claude/memory/memory.db
Embedding mode: lembed (all-MiniLM-L6-v2, 384-dim)
Total memories: 47
Agents: pm (12), tech-lead (18), ic5 (9), ic4 (8)
```
Field values reflect actual DB state.

---

### AC-4: Migration from `.md` Files

**4.1** Migration runs automatically as the final step of `/init-team` (after DB
init and binary download). It can also be triggered standalone via
`/init-team --migrate-only`.

**4.2** Migration reads all files matching
`$MROOT/.claude/memory/<agent>/{cortex,memory,lessons}.md` and inserts each as
a row in `memories` with the correct `agent` and `type` values derived from the
directory name and filename respectively.

**4.3** `context.md` files are NOT migrated. They remain as `.md` files.

**4.4** Migration is idempotent: a duplicate is defined as matching `agent` +
`type` + `content` (exact string). Duplicate rows are not inserted on re-run.

**4.5** After all inserts succeed, migration validates that the count of
successfully inserted rows (new, non-duplicate) equals the count of source
`.md` files processed. If the counts do not match, migration prints the names
of any files that failed and exits non-zero without deleting originals.

**4.6** If validation passes, original `.md` files (`cortex.md`, `memory.md`,
`lessons.md`) are deleted. `context.md` files are not touched.

**4.7** If the delete step fails for any file (e.g., permissions), the file is
listed in a warning and the migration exits non-zero. The DB rows remain (no
rollback).

**4.8** At the end of migration, `/init-team` prints a summary:
```
Migration complete: 24 files â†’ 24 rows inserted, 24 originals deleted.
```
Or on partial failure:
```
Migration: 23/24 files migrated. Failed: .claude/memory/ic4/memory.md (delete failed).
```

### AC-5: `memory-store` Agent Skill

**5.1** The skill signature accepts:
- `agent` (string, required): agent name
- `type` (string, required): one of `cortex`, `memory`, `lessons`
- `content` (string, required): text to store
- `metadata` (object, optional): arbitrary key-value pairs serialized to JSON

**5.2** Calling the skill inserts one row into `memories`. `created_at` and
`updated_at` are set to the current UTC time in ISO-8601 format
(`YYYY-MM-DDTHH:MM:SSZ`).

**5.3** In `mode=lembed` or `mode=ollama`, the skill generates an embedding for
`content` and inserts a row into `memory_embeddings` with the correct `model`
name, `dimensions` integer, and embedding vector.

**5.4** In `mode=fallback`, the skill appends the content to the appropriate
`.md` file for the agent and type (preserving current file format). A warning
is printed: `[memory-store] Embedding unavailable â€” writing to .md fallback.`

**5.5** Concurrent writes from two agents within the same second do not corrupt
the database. Acceptance test: two processes each insert 10 rows simultaneously;
after both complete, the DB contains exactly 20 new rows with correct content.

**5.6** Content up to 100,000 characters is stored without truncation or error.

**5.7** Passing an invalid `type` value returns a descriptive error and does not
insert any row.

---

### AC-6: `memory_embeddings` Schema for Multi-Model Support

**6.1** `memory_embeddings` stores `model` (TEXT) and `dimensions` (INT)
alongside each embedding vector, so rows from different models coexist.

**6.2** When searching, the system queries only rows where `model` matches the
currently active model name (from `memory_config`).

**6.3** On `/init-team --refresh`, if the active mode changes (e.g., lembed â†’
ollama), the system begins a lazy re-embedding pass: any `memories` row that has
no embedding row for the new model is queued for re-embedding. Re-embedding
processes up to 50 rows per invocation to avoid blocking.

**6.4** Old embedding rows (from the previous model) are retained and not
deleted. Search continues to work on new model rows as they accumulate.

**6.5** A `memories` row with zero embedding rows for the current model is
still returned by `memory-recall` in keyword fallback order (appended after
semantic results, labeled `[not yet embedded]`).

---

### AC-7: `memory-recall` Agent Skill

**7.1** The skill accepts:
- `query` (string, required): the search query
- `agent` (string, optional): filter to a single agent's memories
- `type` (string, optional): filter to one memory type
- `limit` (integer, optional, default 5): max results to return

**7.2** With no `agent` filter, the skill searches ALL agents' memories.

**7.3** In `mode=lembed` or `mode=ollama`, results are ranked by cosine
similarity between the query embedding and stored embeddings. Only embeddings
matching the current model are considered.

**7.4** In `mode=fallback`, the skill performs a case-insensitive substring
search across `.md` files and returns matching lines with 2 lines of context.

**7.5** Each result includes: `agent`, `type`, `content`, `score` (float 0â€“1,
semantic mode) or `null` (fallback mode), `created_at`.

**7.6** Zero results returns an empty list, not an error.

**7.7** A query with only stopwords or whitespace returns an empty list, not an
error.

---

### AC-8: `/memory-search` User Command

**8.1** The command is invoked as `/memory-search <query>`. It calls
`memory-recall` internally.

**8.2** `/memory-search` and `/mem-search` are both functional after this change.
`/mem-search` retains its existing keyword-grep behavior unchanged.

**8.3** The output header identifies the active mode:
```
MEMORY SEARCH: "what did we decide about auth?"  [semantic / lembed]
```
or `[semantic / ollama]` or `[keyword / fallback]`.

**8.4** In semantic mode, results are sorted by score descending. Score is
displayed as a percentage rounded to the nearest integer (e.g., `87%`).

**8.5** In fallback mode, output format is identical to `/mem-search`.

**8.6** Each result shows: agent name, memory type, score (semantic mode only),
creation date, and a content snippet (first 200 characters, truncated with `â€¦`
if longer).

**8.7** Running `/memory-search` with no query argument (or only whitespace)
prints usage:
```
Usage: /memory-search <query>
       /memory-search --status
```

**8.8** Running `/memory-search --status` prints the status block defined in
AC-3.7. No search is performed.

---

### AC-9: Fallback Behavior (Runtime)

**9.1** If `memory.db` does not exist (e.g., `/init-team` was never run),
`memory-store` writes to `.md` files as today. No user-visible error beyond the
fallback warning in AC-5.4.

**9.2** If `memory.db` exists but a sqlite extension fails to load at runtime,
the system logs `[memory] Extension load failed: <error>. Switching to fallback
mode.` to stderr and continues in `mode=fallback` for the remainder of the
session. It does not crash or surface an unhandled exception.

**9.3** Data written in fallback mode is never silently dropped. Every
`memory-store` call in fallback mode produces a visible, readable `.md` file
entry.

**9.4** On the next `/init-team` (or `--migrate-only`), fallback-generated `.md`
files are migrated into the DB per AC-4.

---

### AC-10: Non-Functional Requirements

**10.1** A semantic search (`memory-recall`) over 10,000 stored embedding rows
completes in under 500ms on Linux x86_64 with a modern laptop-class CPU.

**10.2** Total disk footprint of extensions + GGUF model is under 50MB.

**10.3** No agent operation (`memory-store`, `memory-recall`) requires outbound
network access after `/init-team` completes in `mode=lembed`.

**10.4** `memory.db`, the `extensions/` directory, and the `models/` directory
are all excluded from git via `.gitignore` (AC-1.7 and AC-2.6).

**10.5** The implementation adds no build step to the plugin. All new
functionality is delivered as markdown skill/command files plus shell scripts
invoked at runtime.

## Out of Scope (MEM-001)

- Global cross-project memory DB (`~/.claude/memory.db`)
- HTTP daemon for memory access
- Docker image distribution
- Working/long-term memory tiers and `/memory-distill`
- Session recording (claude-mem style)
- UI or dashboard
- Native Windows support (WSL2 = Linux, supported)
- Embedding quality benchmarks
- Standalone re-embed command (may be added as v1.1)
